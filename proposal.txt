Goal/Objective
This project evaluates how three distributed fine-tuning strategies—PyTorch Distributed Data Parallel (DDP), DeepSpeed ZeRO Stage 2, and PyTorch Fully Sharded Data Parallel (FSDP)—impact the full lifecycle of large language model deployment, from multi-GPU training to inference with vLLM. While prior work often focuses primarily on training speed and memory efficiency, this study investigates the broader, end-to-end implications of each strategy, including checkpoint structure, export complexity, and deployment compatibility.
DDP serves as a baseline data-parallel approach in which each GPU maintains a full model replica and synchronizes gradients via all-reduce. DeepSpeed ZeRO-2 reduces memory redundancy by partitioning optimizer states and gradients across devices, enabling larger models to be trained within fixed memory budgets. FSDP extends this idea further by sharding model parameters, gradients, and optimizer states, maximizing memory efficiency but increasing checkpoint complexity.
Experiments will be conducted on two model sizes (~1B and ~3B parameters) to analyze how scaling the model size affects training throughput, GPU memory utilization, checkpoint consolidation overhead, and ease of integration with vLLM for inference serving. By systematically comparing these strategies for different model sizes, this project aims to provide practical guidance for selecting distributed fine-tuning approaches when the ultimate goal is efficient and production-ready model deployment.
Challenges
There will be several key challenges involved with undertaking this project. For example, we must ensure a fair comparison: same model architecture, dataset, and target validation loss across all methods. Additionally, we must handle different checkpoint formats (full vs sharded) and make them compatible with vLLM. We must also measure subtle but important aspects like export overhead, vLLM load time, and any precision-related quirks that might affect inference quality or stability.



Approach and Performance Optimization Techniques to be Used
The approach for our study will follow the format described below:
Train the same ~1B and ~3B LLMs on a fixed dataset with:
Baseline: PyTorch Distributed Data Parallel (DDP).
DeepSpeed ZeRO-2 (optimizer + gradient sharding).
PyTorch Fully Sharded Data Parallel (FSDP) (parameter + optimizer sharding).
For each method and model size, measure:
Training metrics: tokens/sec, wall-clock time to reach a fixed validation loss, peak GPU memory.
Export overhead: time to convert training checkpoints to a vLLM-loadable format, peak CPU RAM, and extra code/scripts required.
Inference metrics (via vLLM): model load time, first-token latency, steady-state tokens/sec for a standard prompt distribution.
Implementation Details
Our hardware will require a single multi-GPU node (e.g., 4× modern NVIDIA GPUs with ~24GB VRAM each). We will utilize Columbia’s compute cluster for this part of the project. All training methods (DDP, ZeRO-2, FSDP) will run on the same node configuration. For the inference portion of the project, vLLM serving will be done on 1–2 GPUs from the same class, to emulate a realistic deployment scenario.

Our software stack will be as follows: the primary framework will be PyTorch for model definition and training. Distributed training will then be performed with PyTorch DDP, DeepSpeed for ZeRO-2, and PyTorch FSDP APIs for fully sharded data parallel training. Inference will be done using vLLM for high-throughput and OpenAI-compatible serving. We will use the PyTorch profiler and basic GPU/CPU utilization logging. And the dataset will be a public instruction-tuning dataset (to be determined), with the same train/validation split and preprocessing across all experiments.
